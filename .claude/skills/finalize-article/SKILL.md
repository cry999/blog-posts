# Zenn ブログ記事の最終仕上げ

書き上がった（またはブラッシュアップ済みの）記事を公開前に点検し、リリースできる状態に整えるスキル。
記事全体の整合性チェック・誤字脱字チェックを行い、内容を要約してユーザーにリリースの可否を確認する。許可が出たら公開フラグ等を整え、**「準備完了」を報告して終わる**。

**重要**: このスキルは git の commit / push や公開操作そのものは行わない。最終的な公開（main への push）はユーザーが自分で実施する。スキルの役割は「いつでも push できる状態」を作り、その旨を報告するところまで。

ブラッシュアップの壁打ちをしたい場合は `refine-article`、叩き台を作る段階なら `draft-article` を使う。本スキルは記事がほぼ仕上がっている前提の最終工程。

## 工程

### 1. 対象記事を特定する

どの記事を仕上げるかを確認する。

- ユーザーがファイルを指定していればそれを使う。
- 指定がなければ `articles/` を一覧し、`published: false` のものや直近編集されたものを候補として提示し、選んでもらう。
- 対象ファイルを読み込み、frontmatter と本文を把握する。

### 2. 主題の一貫性をチェックする

まず「この記事が一番伝えたいこと（主題）」を 1 文で言い切れるかを確認する。言い切れないなら主題がぼやけているサイン。主題を軸に以下を点検し、ブレている箇所（`file_path:line`）を示す。

- **主題が一本通っているか**: 冒頭（タイトル・導入・TL;DR）で約束した主題と、本文・まとめで実際に語っている主題が一致しているか。途中で主題がすり替わっていないか。
- **各セクションが主題に奉仕しているか**: それぞれの見出しが主題の理解に貢献しているか。主題と関係の薄い寄り道・蛇足になっていないセクションはないか。
- **主題に対する結論の収束**: まとめ・結論が、冒頭で立てた主題の問いにちゃんと答えているか。約束したのに回収されていない論点がないか。
- **主題のブレを生む記述**: 「ついでに」書かれた別テーマの話が主題をかすませていないか。複数の主題が混在しているなら、絞るか記事を分けるかをユーザーに相談する。

主題からの逸脱を見つけたら、削る・主題に紐づけ直す・別記事に切り出す、のどれが良いかを提案し、ユーザーの判断を仰ぐ。

### 3. 全体の整合性をチェックする

記事を通読し、内容の一貫性を確認する。問題があれば箇所（`file_path:line`）を示して指摘する。

- **タイトルと内容の一致**: タイトルが本文の内容・結論と合っているか。
- **見出し構成の流れ**: セクションの順序が自然か、話の飛躍や抜けがないか、見出しの粒度が揃っているか。
- **主張と根拠の整合**: 結論・主張に根拠や具体例が伴っているか。前半の前提と後半の記述が矛盾していないか。
- **重複・冗長**: 同じ説明の繰り返しや不要な脱線がないか。
- **導入とまとめの呼応**: 冒頭で提示した狙い・課題が、まとめで回収されているか。
- **やり残し**: 「要確認」「TODO」「（ここに〜）」など叩き台時代のプレースホルダや書きかけが残っていないか。残っていれば必ず洗い出してユーザーに知らせる。

### 4. 誤字脱字・表現をチェックする

文章面を点検する。直すべき箇所を具体的に挙げる。

- **誤字脱字・変換ミス**: タイプミス、誤変換、助詞の重複・欠落。
- **表記ゆれ**: 用語・固有名詞・送り仮名・英数字の全半角などの不統一。
- **文体の統一**: ですます調／である調が混在していないか。
- **読みにくさ**: 一文が長すぎる箇所、係り受けの曖昧さ。
- **専門用語**: 初出の用語に説明があるか（対象読者を踏まえて過不足を判断）。

修正は、明らかな誤りはまとめて提案し、ユーザーの合意を得てから `Edit` で反映する。意味や主張に関わる書き換えは勝手にやらず確認する。

### 5. frontmatter を検証する

公開に必要な frontmatter が正しく揃っているか確認する。CLAUDE.md の規約に従う。

- **title**: 設定済みで内容と一致しているか。
- **emoji**: サムネ用の絵文字が 1 文字設定されているか。
- **type**: `tech`（技術記事）か `idea`（アイデア）か、内容に合っているか。
- **topics**: タグが妥当か（Zenn はおおむね 5 個まで）。題材に対して過不足ないか。
- **published**: この時点ではまだ `false`（公開操作は工程 8 で行う）。
- **published_at**: 予約公開する場合のみ。形式は `YYYY-MM-DD HH:MM`。指定が無ければ push 時刻が公開時刻になる。

不足・不適切な項目があれば提案して直す。

### 6. リンク・画像・コードを確認する

形式面の最終チェック。

- **画像参照**: 本文の `![](/images/...)` が指すファイルが実在するか確認する。`draft-article` で作った drawio 図がある場合、`.drawio` がエクスポート済み（PNG/SVG が `images/<slug>/` にある）か、参照パスと一致するかを確認する。未エクスポートなら `drawio --export --format png --output <out>.png <in>.drawio` で書き出す。
- **リンク**: 記事内リンク・外部リンクの記法が壊れていないか。明らかにおかしい URL があれば指摘する。
- **コードブロック**: 言語指定（` ```go ` 等）があるか、閉じ忘れがないか。サンプルコードに明らかな誤りがないか（断定できなければ「要確認」として残す）。
- **未追跡ファイル**: 画像など、公開に必要なファイルが git 管理に入っているか（`git status` で確認し、add 漏れがあればユーザーに知らせる。add 自体は工程 8 の判断に委ねる）。

### 7. 内容を要約し、リリースの許可を求める

ここまでのチェック結果を踏まえ、ユーザーに最終確認を仰ぐ。以下を簡潔にまとめて提示する。

- **記事の要約**: タイトル、対象読者、何を伝える記事か（主題）、主要な見出しの流れを数行で。
- **チェック結果**: 工程 2〜6 で見つけて直した点、および残っている懸念（要確認箇所・未解決の指摘・主題のブレ）があれば明記する。懸念ゼロなら「特になし」と言い切る。
- **公開設定の確認**: 即時公開か予約公開か（`published_at` を使うか）。

そのうえで「この内容でリリース（公開）してよいか？」を明確に尋ねる。`AskUserQuestion` を使ってもよい。**懸念が残っている場合は、それを解消するか、了承の上で進めるかをユーザーに判断してもらう**。許可が出るまで工程 8 には進まない。

### 8. 公開準備を整えて「準備完了」を報告する

リリース許可が出たら、公開できる状態に整える。**push はしない。** 整えるのは以下まで。

1. **frontmatter を公開設定にする**: `published: true` に変更する。予約公開なら `published_at` を設定する。
2. **状態を確認する**: `git status` で、公開に必要な変更（記事 Markdown・画像など）が揃っているか確認する。
3. **準備完了を報告する**: 次の内容を伝えてスキルを終える。
   - 仕上げた記事のファイルパス。
   - `published: true`（予約なら `published_at` も）に設定済みであること。
   - これから push すべき変更内容（変更・追加されたファイル一覧。git status の要約）。
   - 「あとは main へ push すれば公開されます」という案内。**実際の commit / push はユーザーが行う**。

## 注意

- git の commit / push・公開操作は絶対に自動で行わない。準備が整ったことを報告するに留める。
- 本文の意味・主張に関わる書き換えはユーザーの合意を取ってから。誤字脱字など明白な修正はまとめて提案して反映する。
- 不確かな技術的事実は断定せず「要確認」として残し、要約時に懸念として必ず伝える。誤った内容のまま公開準備を整えない。
- リリース許可が出ていないのに `published: true` にしない。許可（工程 7）→ 準備（工程 8）の順を守る。
