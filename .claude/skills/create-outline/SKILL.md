# Zenn ブログ記事のアウトライン作成

このリポジトリの規約（CLAUDE.md 参照）に沿って、Zenn 記事のアウトライン（見出し構成）を組み立てる。
本文は書かない。記事を書き始める前の「設計図」として、見出しの並びと各セクションで扱う中身の要点を固めることが目的。

ドラフト（叩き台の Markdown ファイル）まで作りたい場合は `draft-article` スキルを使う。本スキルはその前段の構成検討に特化する。

## 工程

以下を順に行う。ヒアリング（工程 1〜3）は可能なら `AskUserQuestion` ツールでまとめて尋ねると速い。ただし題材と要点は自由記述になりやすいので、対話で聞き出してもよい。

### 1. 題材について尋ねる（必須）

何についての記事かを尋ねる。例: 「AtCoder ABC458 E の解説」「Go の context の使い方」など。
題材が曖昧なら、対象読者や記事の狙い（何を伝えたいか）まで踏み込んで具体化する。

### 2. 盛り込みたい要点を 3 つ尋ねる（必須）

記事で必ず伝えたいポイントを 3 つ挙げてもらう。
ユーザーが 3 つ思いつかない場合は、題材から候補を提案して一緒に決める。この 3 点がアウトラインの軸になる。

### 3. 記事の種類（テンプレート）を選ぶ

アウトラインのテンプレートを選ぶため、記事がどの系統かを確認する。
`.claude/article-templates/` にテーマごとのテンプレート（1 テーマ 1 ファイル）があるので、まずそれを読んで候補とする。各ファイルの frontmatter の `theme` / `aliases` / `description` を見て、題材に合うものを選ぶ。ユーザーが特にこだわらなければ、題材から妥当なテンプレートをこちらで判断して進める。

合うテンプレートが無ければ、その場で簡易に構成を組んでもよい。よく使う系統なのにテンプレートが無い場合は、`manage-outline-templates` スキルで新規テンプレート化することを提案する。

参考として、初期テンプレートには以下がある（実体は `.claude/article-templates/` を参照）:

- **AtCoder 問題解説系**（`atcoder`）: この問題について / 問題概要 / 考察 / 解法・実装 / 計算量・別解 / この問題の面白いところ / まとめ
- **技術解説・競プロ系**（`tech-competitive`）: 問題概要 / 考察・アプローチ / 実装 / 計算量・補足 / まとめ
- **ノウハウ・チュートリアル系**（`howto-tutorial`）: 背景・課題 / 結論（先に要点） / 詳細解説 / 注意点 / まとめ
- **アイデア・考察系**（`idea-essay`）: 問題提起 / 主張 / 反論への応答 / まとめ

### 4. アウトライン（見出し構成）を組み立てて提示する

選んだテンプレートをベースに、要点 3 つを軸にセクション構成を組み立てる。
各見出しには「そのセクションで何を書くか」を 1〜2 行で添える。要点 3 つは確実にどこかのセクションに反映する。
出力は Markdown の見出し（`##` / `###`）と短い説明文の形で提示する。本文は書かない。

提示例:

```markdown
## 問題概要
- ABC458 E がどんな問題かを 1 段落で要約する
- 制約（N の上限など）に触れる

## 考察・アプローチ
- 要点①: 〇〇に着目すると区間の数え上げに帰着できる
- なぜその発想に至るかを補足する

## 実装
- 要点②: 累積和を使った実装方針

## 計算量・補足
- 要点③: 全体の計算量が O(N log N) であること

## まとめ
- 学びを 2〜3 行で振り返る
```

### 5. 合意を取り、次のアクションを案内する

提示したアウトラインにユーザーの合意を取る。修正要望があれば反映して再提示する。
合意が取れたら、このアウトラインを元に `draft-article` スキルで叩き台 Markdown を生成できることを案内する。

## 注意

- アウトラインの段階なので本文は書かない。あくまで見出しと各セクションの要点メモにとどめる。
- 不確かな技術的事実（計算量・API 仕様など）は断定せず、「要確認」と明記する。誤った前提でアウトラインを固めない。
- 要点が 3 つに収まらない／多すぎる場合は、セクションの粒度を調整してユーザーと相談する。
